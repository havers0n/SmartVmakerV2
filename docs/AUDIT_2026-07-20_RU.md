# Scrimspec: технический и продуктовый аудит

Дата: 20 июля 2026. Объект: текущая рабочая копия репозитория, включая незакоммиченные файлы. Код, миграции и конфигурация не менялись. Проверка не выполняла внешних API-вызовов, миграций или операций с удалёнными данными.

## 1. Executive Summary

**Что это сейчас.** Scrimspec — ранний внутренний прототип из двух частично пересекающихся продуктов:

1. **Discovery для YouTube**: ниша → набор поисковых запросов → синхронный поиск YouTube → сохранение видео/каналов → эвристические opportunity/outlier/cohesion-метрики → title-embedding/clustering → CSV-экспорт кандидата. Этот контур находится в `apps/dashboard/src/app/discovery`, `src/server/discovery-runs.ts`, миграциях `0013`–`0023`.
2. **HWAR/generation**: зачатки генерации сценариев, шаблонов/beat-ов, keyframe и MiniMax animation jobs. Здесь есть UI и SQL-очереди, но core facade отключает либо подменяет stub-ами большую часть подсистем, а production build не проходит.

Это **не** единая система Content Format Intelligence и не production-ready production pipeline. Реального контура «транскрипт → структурированный анализ → доказательства → сохраняемый Content Format → сценарий/сцены/активы» в коде нет. Нынешний `Content Format` — это вычисляемые из заголовков и метаданных поля `discovery_clusters`, а не самостоятельная доменная сущность.

**Сильнейшие части.** Есть осмысленная реляционная база YouTube-метаданных, дедупликация по YouTube ID, Discovery Run с результатами и сырыми evidence rows, приличный набор детерминированных эвристик и unit-тестов discovery, отдельные SQL-очереди с idempotency-индексами для старого generation-контура, R2 presigned URL с owner-check, HMAC/SSRF-проверки webhook.

**Главные ограничения.** Discovery исполняется внутри POST-запроса без checkpoint/resume, квотного budget-а, отмены и фонового worker-а; discovery/seed/niche данные не имеют tenant ownership и endpoint-ы не делают object authorization. Схема не содержит transcript, provenance AI-вызовов, reusable Content Format, research/script/scene/media-generation модели. Build, typecheck, lint и полный test run не зелёные. Рабочая копия содержит незакоммиченный discovery-контур и миграции, что делает состояние развёртывания неуправляемым.

**Главный технический вывод.** Не переписывать всё: сохранить discovery data spine, YouTube hydration, scoring/evidence и generation queue primitives. Следующий этап должен сделать один надёжный вертикальный контур **Discovery Run → отобранный набор видео → асинхронная обработка → версионированный Content Format с evidence**, а не добавлять ещё UI/провайдеров генерации.

## 5. End-to-End Scenarios

### A. Discovery run

**Работает.** UI POSTs `/api/discovery-runs`; `createDiscoveryRun` creates status `running`, loops enabled queries × orders, calls YouTube, upserts channels/videos and `video_discoveries`, calculates intelligence/clusters, then marks completed. Results, channels, opportunity metrics and candidates read from persisted evidence. Paths: `app/discovery/page.tsx`, `runs/[runId]/page.tsx`, `api/discovery-runs/*`, `server/discovery-runs.ts:1551+`, `shared/lib/youtube.ts`.

**Обрыв/ручное.** Request remains open for the entire external workload; hydration is a separate manual POST. There is no queued job, progress/checkpoint per query/page, resume/cancel, dedupe across a retry beyond DB constraints, quota budget, timeout policy or run-level idempotency key. A failed request marks run failed but does not record completed query/search-order checkpoints. CSV only exports a selected cluster, not a full run.

**Готовность:** H for small trusted internal searches only; not reliable for large or unattended runs.

### B. Перспективный Content Format

**Работает.** Videos are grouped into `discovery_clusters`; deterministic title/token, channel, VPD and language signals produce candidate scores. Candidate cards carry representative videos/channels; curation excludes rows and CSV exports preserved evidence. Migrations `0021`–`0023` add semantic/title embeddings and format-shaped fields.

**Обрыв.** No independently versioned/approved `ContentFormat`, no relation Format→source evidence outside a mutable cluster, no transcripts, semantic similarity is title-derived, no user save-to-project or research workspace. Fields named hooks/emotions/visual style are inferred from metadata/title patterns; migration `0022` explicitly says no transcript/media analysis. The UI shows `Send to Research (coming soon)`.

**Готовность:** P/X; useful research ranking hypothesis, not a defensible Content Format product.

### C. Анализ видео

**Работает.** Existing videos can be listed; `startAnalysis` enqueues unique `(video, analyzer)` rows and analysis worker/core handler conceptually stores `analysis_results`. `/api/videos/[id]/analysis` exists and is admin-gated.

**Обрыв.** No transcript retrieval/storage, no source content normalization, no prompt/version/model/cost capture, no typed structure for hooks/scenes/patterns, no multi-video comparison. The run fails static build at `analysis/[id]/page.tsx:147`; analysis handler unit suite has 8 failures. This makes the visible analysis UI insufficient evidence of operation.

**Готовность:** P, and not safe as a user-facing intelligence feature.

### D. Переход к производству

**Работает.** A separate generation path can create template/beat-based projects, queue keyframes, call selected image models and submit MiniMax image-to-video/first-last tasks. Animation webhook can validate signature, restrict direct URL hosts, upload result to R2 and create an asset.

**Обрыв.** It is not linked from discovery formats/videos. Script is largely `project.meta`; pipeline status contains explicit placeholder logic. `approve-script` makes an empty `storageUrl` asset. Core facade returns disabled stubs for ingest/generation/animation/assets even when flags are enabled. No asset versioning, storyboard, voice/music/editing/publish, approval history or final composition pipeline.

**Готовность:** X/P: useful technical foundation for keyframes/animation only, not a production pipeline.

## 6. Data Model Audit

### Existing core

* **Channel/Video:** `youtube_channels` (`youtube_channel_id` unique, metadata/subscriber fields) and `youtube_videos` (`youtube_id`, URL uniqueness, channel FK). This is a usable shared catalog, but it is global rather than tenant-scoped.
* **Discovery:** `niches` → `niche_queries`; `discovery_runs` → `video_discoveries` (run/query/video evidence); `seed_sources` → `niche_candidates`; `discovery_clusters` → `discovery_cluster_videos`; `discovery_video_embeddings`. This is the strongest new data spine.
* **Generation:** `aes_core.story_templates` → `beats`; `generation_pipeline.generation_projects` → `assets`; four `jobs.*_job_queue` tables; plus `generation_animation_jobs`, legacy `generation_events/tasks/batches/clips`.
* **Analysis:** `analysis_results` and `analysis_job_queue` attach to `youtube_videos`; output is generic `aes_breakdown` JSON plus score/tags, without source/evidence granularity.

### Missing or inadequate future concepts

| Target concept | Current substitute | Limitation |
|---|---|---|
| Content Format | `discovery_clusters` fields | Mutable run-local cluster, no lifecycle/owner/version/provenance/approval. |
| Transcript | none | Cannot evidence spoken hooks, sections or claims. |
| AI Extraction / Analysis Result | `analysis_results.aes_breakdown` JSON | No input hash, prompt/model/version, schema version, chunk/source citations or cost. |
| Evidence/Source | `video_discoveries`, cluster membership | Only discovery provenance; no spans/timestamps/quote/asset provenance. |
| Template | `story_templates`/`beats` | AES template unrelated to discovered format and no ownership enforcement. |
| Script/Scene/storyboard | `generation_projects.meta`, beats | Unstructured, not versioned, not linked to source/format. |
| Media Asset | `assets` | Has storage fields/hash but lacks generic input/output relation, versions and consistent provider job FK. |
| Generation Task | four queues + `generation_animation_jobs` + legacy tasks | Duplicated lifecycle, inconsistent statuses and external IDs. |
| Production Project | `generation_projects` | Owner nullable/no auth FK; no links to idea/format/research/script. |

### Ownership, RLS, lifecycle

`generation_projects.owner_id` and `characters.owner_id` are nullable uuid fields and comments say auth FK is deferred (`schema.ts:314,361`). Discovery entities have no owner/workspace fields. RLS is enabled only for a subset of discovery tables in migrations (`0014`, `0015`, `0018`, `0021`); policies are not a complete tenant model, while application DB reads use direct `pg`/Drizzle and `getSupabaseClient` uses service role. Therefore endpoint authorization, not RLS, is currently the practical boundary—and discovery routes simply trust middleware authentication, then query global IDs.

Job uniqueness is good for `(video, analyzer)`, `(project, scene, frame)`, `(project, scene animation)` and video IDs. Retry metadata exists but status/stage/idempotency are not enforced uniformly. Soft delete exists in selected generation/legacy rows, absent from discovery. There is no restore workflow, no event history and no durable immutable AI provenance.

## 7. Production Risks

### Critical

* **Unusable deploy artifact.** Current `pnpm --filter dashboard build` and `pnpm type-check:all` fail. A standard production deployment cannot be certified from this revision. First build blocker: `apps/dashboard/src/app/analysis/[id]/page.tsx:147`; more errors include missing `@scrimspec/hwar-core` exports and worker date-type mismatches.
* **Cross-user discovery data exposure/modification.** Any authenticated user can enumerate/read/mutate global discovery niches, sources, runs and curation because domain tables lack owner/workspace and routes such as `api/discovery-runs/[id]/*`, `api/niches/*`, `api/seed-sources/*` do no object authorization. Middleware only authenticates.

### High

* **Discovery can consume quota/cost without recovery.** `createDiscoveryRun` does all external search, persistence and intelligence synchronously. Browser/server timeout or process restart loses progress semantics; retry repeats calls; no budget/concurrency/backpressure/cancel exists.
* **False production pipeline.** Core flags do not enable real ingest/generation/animation/assets (`packages/hwar-core/src/index.ts:174–194` returns disabled stubs). The generation UI falls back to mock scenarios on missing/failed MiniMax. Users can interpret a successful-looking UI as actual generated work.
* **Queue operational inconsistency.** Workers each poll SQL manually; generic `runJobTick` exists but is not universally used. Cleanup resurrects all `processing` rows based on `updatedAt`, with no lease owner or external task reconciliation. Duplicate/late webhook and job paths use two animation schemas.
* **Insecure TLS setting observed during build.** Node emitted that `NODE_TLS_REJECT_UNAUTHORIZED=0` is set in the environment. It is not in committed code, but it makes any outbound HTTPS in that runtime susceptible to MITM and must not be part of deployment.

### Medium

* No prompt injection policy/sanitization boundary for YouTube titles, source notes, future transcript/web input. Current prompts interpolate seed fields directly (`seed-sources.ts`).
* AI calls have no request IDs, input/output hash ledger, token/cost telemetry or model/prompt version; auditability and reuse cannot be guaranteed.
* Webhook has good HMAC and SSRF host allowlist, but in non-production intentionally accepts unsigned payloads if secret absent (`api/webhooks/halu/route.ts`).
* Lint is not configured: `next lint` enters an interactive setup wizard; many packages merely echo success. It cannot protect CI.
* Backup docs expose a concrete project identifier/host and state historic migration count inconsistent with current 24 journal entries; no verified restore exercise or retention automation is in code.

### Low

* Outdated `caniuse-lite`/baseline warnings and Edge-runtime Supabase Node API warnings during build.
* Corrupted mojibake comments/docs make operational instructions hard to trust.
* Root repository URL still refers to `SmartVmakerV2`, not Scrimspec (`package.json`).

## 8. Dead Code and False Completeness

* `packages/hwar-core/src/index.ts`: enabled flags for ingest, generation, animation and assets return disabled stubs; public facade is not the same system as direct dashboard workers.
* `packages/workers/src/index.ts` is explicitly a placeholder; `packages/workers` test script echoes `No tests yet`, despite three test files existing outside the invoked package script.
* `packages/api-client/src/index.ts` lists `/api/hwar/*` operations for routes that do not appear in `app/api`; it is an aspirational client, not a contract.
* Discovery candidate UI contains `Send to Research (coming soon)` backed only by a literal placeholder field; no research handoff exists.
* `generation.ts` returns mock scenarios on absent/failed MiniMax; this is intentionally development behavior but is not visibly distinguished as synthetic product output.
* `approve-script` creates an `assets` row with empty `storageUrl`; it demonstrates pipeline wiring, not a valid media artifact lifecycle.
* Multiple animation implementations coexist: legacy `animation_job_queue` + worker/webhook and newer `generation_animation_jobs` via `DefaultAiRouter`; neither is the single source of truth.
* `shorts-research` and `tools/yt-orchestrator-python` include real-looking CSV/transcript artifacts and scripts but have no app/runtime integration; treat as research material only.
* Migration `0022` format fields can make the UI look like deep multimodal format analysis, while its own SQL says it stores no transcript or media analysis.

## 9. What can be the foundation

1. **Discovery relational spine** (`niches`, `niche_queries`, `discovery_runs`, `video_discoveries`, `youtube_*`). It stores raw search evidence, preserves run/query/order/position and deduplicates YouTube records. Stabilize by adding workspace ownership, immutable run checkpoints and a worker-backed orchestration layer.
2. **YouTube normalization and channel hydration** (`shared/lib/youtube.ts`, `hydrateDiscoveryRunChannels`). It already cleanly separates external identifiers from internal records and supports upsert. Fix typecheck, quota/error policy and freshness/cache metadata before scaling.
3. **Deterministic ranking evidence** (`niche-cohesion.ts`, `niche-branching.ts`, `discovery-runs.ts`). Tests cover much of the calculation logic. Preserve it as explainable candidate ranking, but label it as metadata heuristics—not causal format intelligence.
4. **SQL job primitives** (`jobs.*`, `hwar-core/jobs/processor.ts`). Attempts, retry time, lock time, stage and idempotency indexes are a good start. Unify all workloads onto one generic task contract with leases, external task reconciliation and monitoring.
5. **Asset/R2 controls and webhook hardening** (`storage-client`, `api/r2/*`, `api/webhooks/halu`). Signed URLs enforce project ownership; webhook includes HMAC, timing-safe comparison and SSRF controls. Reuse after a consistent asset/task model is chosen.
6. **Provider/model registry** (`aes_core.ai_providers`, `ai_models`). It is the right configuration idea, including model type/defaults/response adapter. Current code bypasses it in Gemini discovery and hardcodes MiniMax fallback, so make it the sole provider configuration surface.

## 11. Recommended next stage

**One goal: turn Discovery into a reliable, evidence-backed Content Format workflow for trusted internal users.**

Concrete outcomes:

1. A worker-backed, resumable Discovery Run with persistent per-query/search-order checkpoint, idempotency, quota budget, cancellation and clear partial-failure UI.
2. A first-class, versioned `content_formats` aggregate created from a curated discovery cluster, preserving immutable video/channel/query evidence and user decision history.
3. A minimal video-intelligence slice: transcript acquisition/storage (where permitted), chunked structured extraction, source timestamps and prompt/model/input-hash ledger—initially for one video and then a selected candidate set.
4. Workspace/owner model plus object authorization for all discovery rows; RLS policies that match the application model instead of relying on global tables.
5. A green engineering baseline: repair static build/typecheck/test failures, non-interactive lint configuration, and CI that runs these checks without credentials.

Why this first: it addresses the product's defining unit (Content Format) and removes the risky synchronous pipeline, while reusing the strongest existing data and algorithms. It creates a stable input contract for later scripts, scenes and MiniMax generation.

Consciously **do not** yet add ComfyUI, new MiniMax modalities, voice/music, final editing, a broad public API, or more polished generation screens. Those multiply cost and state-management complexity before the source intelligence and task model are trustworthy.

## 12. Open Questions

* Is the active production database actually at migration journal entry 23, and are the uncommitted `0016`–`0023` files already applied? The repository cannot prove live schema state.
* Which workers run continuously in production, with what process manager, concurrency, secrets and health supervision? `ecosystem.config.cjs` is development-oriented and excludes cleanup/enrichment.
