### 2. Контекст и Повествование: "Великая Синхронизация"

Наш диалог — это история преодоления фундаментального "раскола миров", который часто случается в проектах на ранней стадии. С одной стороны, у вас было четкое и глубокое Видение Продукта (AEC Studio, Эмоциональная Архитектура, анализ вирусности) и продуманная Целевая Архитектура (Action Runner, Workers, Schema-First). С другой стороны, была Реальность Кодовой Базы — артефакты от прототипов, рассинхронизация между кодом и базой данных, не до конца настроенные инструменты и "белые пятна" в конфигурации.

Наше путешествие можно разделить на три акта:

*   **Акт I: Диагностика и Осознание Хаоса.** Мы начали с анализа документации и кода, и быстро поняли, что система нестабильна. Ключевые проблемы: дублирование кода (старый Express-сервер, две `app` директории) и полная рассинхронизация между Drizzle-схемой, файлами миграций и реальным состоянием БД в Supabase, измененной вручную.

*   **Акт II: Битва с Инфраструктурой (Самый сложный этап).** Это была настоящая война на несколько фронтов, где мы методично, шаг за шагом, побеждали одного "дракона" за другим:
    *   **Дракон SSL/VPN:** Ошибки `SELF_SIGNED_CERT_IN_CHAIN` и `connection forcibly closed`. Мы победили их, поняв, что VPN был корнем зла, и внедрив надежные решения: `rejectUnauthorized: false` для dev-среды и `NODE_TLS_REJECT_UNAUTHORIZED=0` как "команду-вездеход".
    *   **Дракон Сети (IPv6):** Ошибка `ENOTFOUND` / `ENETUNREACH`. Мы победили ее, заставив все части системы (Drizzle, Next.js, Workers) использовать пулер соединений (`DRIZZLE_DATABASE_URL`) и принудительный IPv4 (`family: 4`).
    *   **Дракон Миграций:** Самый коварный враг (`No schema changes`, `file not found`, `schema already exists`). Мы победили его, полностью "сплющив" историю миграций и установив новый, единственно верный рабочий процесс "Schema-First".
    *   **Дракон Конфигурации:** Мы отладили загрузку переменных окружения (`.env`) для всех частей монорепозитория (Next.js, Workers), починили `drizzle.config.ts`, `pnpm-workspace.yaml` и `next.config.mjs`.

*   **Акт III: Рождение Конвейера Данных.** Победив инфраструктурных "драконов", мы смогли построить полный, сквозной MVP-сценарий:
    1.  Создали надежный **Action Runner**.
    2.  Создали и отладили **UI на странице `/ingest` и `/analyze`** (включая пагинацию).
    3.  Расширили **базу данных** для полной поддержки YouTube API.
    4.  Создали и запустили все три воркера: **Ingest, Enrichment и Analysis**.
    5.  Отладили последние баги (формат даты, импорты Drizzle, дубликаты) и увидели заветное `Job completed successfully!`.

Параллельно с этим, был проведен глубокий аудит кода, который дал нам четкий и приоритизированный Backlog для дальнейшего развития.

### 4. Архитектурная Карта Системы

Система построена на базе pnpm-монорепозитория, что обеспечивает четкое разделение ответственности между компонентами и переиспользование кода.

```
/
├── apps/
│   └── dashboard/      # Next.js приложение (Фронтенд + API Layer)
│       └── src/app/
│           ├── (pages)/
│           │   ├── analyze/ # UI для анализа видео
│           │   └── ingest/  # UI для запуска сбора видео
│           └── api/
│               └── actions/
│                   ├── handlers/ # Конкретные обработчики (ingest, analysis)
│                   └── route.ts  # <- "Action Runner" - Единая точка входа API
├── packages/
│   ├── api-client/     # (Будущее) Типизированный клиент для API
│   ├── core-domain/    # Общие типы, Zod-схемы (контракты данных)
│   ├── db/             # Все, что связано с БД (Drizzle ORM)
│   │   └── migrations/
│   │       └── schema.ts # <- "Конституция" - Единственный источник истины для БД
│   └── workers/        # Асинхронные фоновые обработчики
│       ├── ingest-worker.ts
│       ├── enrichment-worker.ts
│       └── analysis-worker.ts
└── package.json
```

#### Ключевой Архитектурный Паттерн (Конвейер Данных):

Весь основной функционал построен по единому асинхронному паттерну, что делает систему предсказуемой и надежной:

**UI → `callAction` → Action Runner (`/api/actions`) → Запись в `*_job_queue` → Worker (`SKIP LOCKED`) → Внешнее API → Запись в `*_results_table`**

1.  **UI (`/analyze`, `/ingest`):** Пользователь инициирует действие (например, "Начать анализ").
2.  **`callAction`:** Фронтенд вызывает типизированную функцию, которая отправляет POST-запрос на единый API-эндпоинт.
3.  **Action Runner:** Центральный маршрутизатор (`/api/actions/route.ts`) принимает запрос, валидирует его и передает управление соответствующему `handler`-у.
4.  **Action Handler:** Обработчик (например, `analysis.startAnalysis`) проверяет входные данные с помощью Zod-схемы из `core-domain`, выполняет быструю логику (например, проверку на дубликаты) и создает новую запись-задачу в соответствующей таблице-очереди (`analysis_job_queue`).
5.  **Worker:** Независимый Node.js процесс (например, `analysis-worker`) постоянно опрашивает свою таблицу-очередь. С помощью `SELECT ... FOR UPDATE SKIP LOCKED` он атомарно захватывает одну свободную задачу, не мешая другим воркерам.
6.  **Внешнее API:** Воркер выполняет "тяжелую" работу: обращается к внешним сервисам (YouTube API, Gemini API).
7.  **Запись результата:** После успешного выполнения (или ошибки) воркер обновляет статус задачи в очереди и записывает результат в основную таблицу (например, `analysis_results`).

### 5. Анализ Схемы Данных (`packages/db/migrations/schema.ts`)

"Конституция" проекта. Схема полностью синхронизирована с Supabase и является единственным источником истины.

#### Основные сущности:
*   **`youtube_videos`**: Центральная таблица, хранящая всю информацию о видео, полученную с YouTube. Имеет уникальный индекс по `youtubeId` для предотвращения дубликатов.
*   **`analysis_results`**: Хранит результаты анализа от ИИ-моделей. Связана с `youtube_videos` через `videoId`.
*   **`jobs.ingest_job_queue`**: Очередь задач для сбора видео. Содержит параметры поиска YouTube API, статус и количество попыток.
*   **`jobs.analysis_job_queue`**: Очередь задач для анализа видео. Содержит ссылку на `videoId`, статус и информацию об ошибках.
*   **`aes_core`, `generation_pipeline`**: Группы таблиц, относящиеся к основной бизнес-логике "Эмоциональной Архитектуры" (шаблоны историй, биты, проекты генерации, ассеты). Они закладывают фундамент для будущего функционала.

### 6. Backlog: Что Предстоит Сделать (Задачи из аудита)

Это приоритизированный список задач для обеспечения промышленной надежности и безопасности системы.

#### P0 (Критично):
1.  **Rate limiting на Redis:** Заменить текущую in-memory реализацию на более надежное решение на базе Redis для защиты от злоупотреблений API.
2.  **Аутентификация API:** Закрыть все эндпоинты, кроме `/api/health`, проверкой сессии пользователя (например, NextAuth.js или Clerk).
3.  **Connection pool таймауты:** Явно настроить `max`, `idleTimeoutMillis`, `connectionTimeoutMillis` в `packages/db/src/client.ts` для предотвращения исчерпания соединений под нагрузкой.

#### P1 (Важно):
1.  **CSP заголовки:** Внедрить строгую Content Security Policy в `middleware.ts` для защиты от XSS-атак.
2.  **Retry-механизм:** Реализовать надежный механизм повторных запросов с экспоненциальной задержкой (exponential backoff) для всех `fetch`-вызовов к внешним API (YouTube, Gemini).
3.  **Кэширование:** Внедрить кэширование ответов от YouTube API на Redis, чтобы снизить затраты и избежать превышения квот.
4.  **Логирование:** Заменить `console.log` на структурированный логгер (например, Pino), чтобы обеспечить централизованный сбор и анализ логов.
5.  **Unit-тесты:** Написать unit-тесты для критически важной логики: Action-хендлеры и основные функции воркеров.

#### P2 (Желательно):
*   **Мониторинг и Оповещения:** Внедрить Sentry для отслеживания ошибок в реальном времени.
*   **CI/CD:** Настроить конвейер непрерывной интеграции и доставки (GitHub Actions) для автоматического тестирования и развертывания.
*   **E2E тесты:** Написать сквозные тесты (Playwright/Cypress) для ключевых пользовательских сценариев.

### 7. Ключевые Паттерны, Правила и "Золотые" Команды

Это "шпаргалка" для любого разработчика, работающего с проектом.

#### Золотое Правило
> Единственный источник истины для схемы БД — **`packages/db/migrations/schema.ts`**. Никогда не изменяйте схему вручную через UI Supabase.

#### Workflow Миграций (Надежный, Ручной)
1.  **Изменить `schema.ts`**: Внести необходимые правки в файл.
2.  **`pnpm --filter db generate`**: Сгенерировать SQL-файл миграции в `packages/db/migrations`.
3.  **Скопировать SQL**: Открыть сгенерированный `.sql` файл, скопировать его содержимое.
4.  **Выполнить в Supabase**: Вставить и выполнить скопированный SQL в **SQL Editor** в панели управления Supabase.
5.  **Зарегистрировать миграцию**: Выполнить в том же SQL Editor команду `INSERT INTO drizzle.__drizzle_migrations (hash, created_at) VALUES ('<имя_файла_без_ext>', NOW());` чтобы Drizzle знал, что миграция применена.

#### Команды-"Вездеходы"
*   **При ошибке SSL/TLS (часто из-за VPN):**
    ```bash
    NODE_TLS_REJECT_UNAUTHORIZED=0 <command>
    ```
*   **При сетевой ошибке `ENOTFOUND` (часто из-за IPv6):**
    *   Убедиться, что используется строка подключения с **пулером соединений** (`DRIZZLE_DATABASE_URL`).
    *   Убедиться, что в `packages/db/src/client.ts` в конфигурации `pg.Pool` прописано `family: 4`.

#### Запуск Системы (4 терминала)
1.  **Фронтенд и API:** `pnpm --filter dashboard dev`
2.  **Воркер сбора:** `pnpm --filter workers dev:ingest`
3.  **Воркер обогащения:** `pnpm --filter workers dev:enrich`
4.  **Воркер анализа:** `pnpm --filter workers dev:analysis`
